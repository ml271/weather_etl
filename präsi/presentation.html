ETL — Extract

Extracting Weather Data

Daily Variables (7 days)

  • Temperature max / min (°C)
  • Precipitation sum (mm)
  • Snowfall sum (cm)
  • Wind speed max / Wind gusts max (km/h)
  • UV Index max
  • WMO weather code
  • Sunrise / Sunset times

Hourly Variables (168 hours)

  • Temperature & Apparent temperature (°C)
  • Precipitation / Rain / Snowfall
  • Wind speed & direction
  • Relative humidity (%)
  • Sunshine duration (sec/hour)
  • Soil temperature: 0 cm / 6 cm / 18 cm
  • Soil moisture: 0–1 / 1–3 / 3–9 cm layers (m³/m³)
XCom payload:  city · latitude · longitude · fetched_at (UTC) · full Open-Meteo JSON response
API: Free tier, no API key required. Timezone set to Europe/Berlin for correct local time strings.
ETL — Transform

Transforming the Raw Data

1
Unpack API Arrays → Flat Row Dicts
Open-Meteo returns columnar arrays (e.g. temperature_2m_max: [12.1, 15.3, ...]). Transform zips all arrays by index into individual row dicts — one dict per calendar day or hour.
2
WMO Code Enrichment
Each row's numeric WMO weather code (e.g. 63) is translated to a human-readable string ("Moderate Rain") using a built-in lookup table covering all 30 WMO codes.
3
Alert Rule Evaluation (YAML Config)
Alert thresholds are loaded from alerts_config.yaml (not hardcoded). Each rule defines: name, severity, conditions (AND logic), message. Triggered alerts are included in the XCom payload alongside daily/hourly records.
Example alert rules: temperature_max > 35 → danger precipitation_sum > 20 → warning wind_speed_max > 60 → danger snowfall_sum > 10 → info
ETL — Load

Loading Data into PostgreSQL

UPSERT Pattern

  • Unique constraint: (city, forecast_date) for daily
  • Unique constraint: (city, forecast_time) for hourly
  • On conflict → update all fields in-place
  • No accumulating duplicates across runs

Alert Lifecycle

  • Before insert: all previous alerts for the city set to is_active = false
  • Fresh alerts from current run inserted as is_active = true
  • Dashboard always shows current forecast state only

User Warning Pipeline

Users configure custom alert rules — the second Airflow DAG checks them hourly and sends email notifications.

Condition Types

  • Parameters: temp max/min, precipitation, snowfall, wind, gusts, UV
  • Comparators: > >= < <= ==
  • Logic: AND across all conditions

Validity Modes

  • date_range — from / to specific dates
  • weekdays — e.g. Saturday & Sunday only
  • months — e.g. January & February only

Pre-built Templates

Heatwave: temp_max > 32°C
Frost: temp_min < 0°C
Heavy Rain: precip > 20mm
Storm: wind > 60 km/h
Snowfall: snow > 10 cm
